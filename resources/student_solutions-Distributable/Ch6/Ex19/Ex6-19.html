Examples of protected methods and fields include the following:
  1. javax.swing.ImageIcon
    1. protected static Component component
    2. protected static MediaTracker tracker
    3. protected void loadImage(Image image)
  2. javax.swing.AbstractButton
    1. protected ActionListener actionListener
    2. protected ChangeEvent changeEvent
    3. protected ChangeListener changeListener
    4. protected ItemListener itemListener
    5. protected ButtonModel model
    6. protected int checkHorizontalKey(int key, String exception)
    7. protected int checkVerticalKey(int key, String exception)
    8. protected void configurePropertiesFromAction(Action a)
    9. protected ActionListener createActionListener()
    10. protected PropertyChangeListener createActionPropertyChangeListener(Action a)
  3. java.lang.Object
    1. protected Object clone()
    2. protected void finalize()

A hostile child class of one of these classes would be able to modify the protected fields in an instance of itself, however it would not be able to modify these fields in an instance of any other child of the parent. Protected fields are also visible within the package, however the hostile code cannot add itself to the package and so it can't attack in this way either. Thus, the fields are safe from attack.

The primary difficulty with protected fields is not modification by hostile code, but dependencies between parent and child classes that occur when the child directly accesses the protected fields. This increases coupling between the child and its parent and makes it more difficult in the future to modify the parent.
